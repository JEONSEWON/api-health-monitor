## 당신의 API 모니터링은 충분히 안전한가요?

대부분의 API 모니터링 도구는 이렇게 작동합니다.

> "HTTP 200 OK가 왔다 → 정상"

하지만 이것만으로는 충분하지 않습니다. **HTTP 200을 반환하면서도 응답 데이터가 완전히 망가진 경우**가 실제 프로덕션에서 빈번하게 발생합니다. 이를 **Silent Failure(사일런트 페일러)**라고 합니다.

이 글에서는 Silent Failure가 무엇인지, 어떤 상황에서 발생하는지, 그리고 실제로 어떻게 탐지하는지 코드 예시와 함께 설명합니다.

---

## Silent Failure란?

Silent Failure는 다음 조건을 모두 만족하는 상태입니다.

- HTTP 상태 코드: **200 OK** ✅
- 응답 시간: **정상 범위** ✅
- 실제 서비스: **망가짐** ❌

예를 들어 이런 상황입니다.

```json
// 정상 응답
{
  "status": "ok",
  "data": {
    "users": [{"id": 1, "name": "John"}],
    "total": 1
  }
}

// Silent Failure — HTTP 200이지만 데이터 없음
{
  "status": "ok",
  "data": {
    "users": [],
    "total": 0
  }
}
```

두 응답 모두 HTTP 200을 반환하지만, 두 번째는 DB 연결 오류나 쿼리 버그로 인해 빈 결과를 반환하고 있습니다. 전통적인 업타임 모니터링은 이 차이를 감지하지 못합니다.

---

## 실제 발생 사례

### 1. 결제 API 오동작

```
GET /api/checkout/status → HTTP 200

응답: {"status": "processing", "payment_id": null}
```

`payment_id`가 null인데도 200을 반환. 사용자는 결제가 진행 중이라고 생각하지만 실제로는 결제가 처리되지 않고 있습니다.

### 2. 검색 API 빈 결과

Elasticsearch 연결이 타임아웃 났지만 애플리케이션이 빈 배열을 반환합니다.

```json
{
  "results": [],
  "total": 0,
  "took_ms": 0
}
```

사용자는 검색 결과가 없다고 생각하고, 엔지니어는 알림을 받지 못합니다.

### 3. 인증 토큰 만료

```json
{
  "success": true,
  "token": null,
  "message": "Token generation failed"
}
```

HTTP 200이지만 토큰이 null. 이후 모든 인증된 요청이 실패합니다.

## 탐지 방법 3가지

### 방법 1: Keyword Assertion (키워드 체크)

가장 간단한 방법입니다. 응답 바디에 특정 문자열이 **있어야 하는지** 또는 **없어야 하는지** 검사합니다.

**예시 설정:**
```
Keyword Present: "payment_id"
Keyword Absent: "error"
```

**적합한 상황:**
- 응답 바디가 단순한 경우
- JSON이 아닌 HTML/텍스트 응답 체크
- 빠른 설정이 필요할 때

**Python으로 직접 구현하면:**
```python
import requests

def check_keyword(url: str, keyword: str, should_present: bool = True) -> bool:
    response = requests.get(url, timeout=10)
    if response.status_code != 200:
        return False
    
    keyword_found = keyword in response.text
    return keyword_found == should_present

# 사용 예시
is_healthy = check_keyword(
    "https://api.example.com/health",
    keyword="payment_id",
    should_present=True
)
```

---

### 방법 2: Regex Pattern Matching

정규식으로 응답의 패턴을 검사합니다. Keyword보다 유연하게 사용할 수 있습니다.

**예시 패턴:**
```regex
# UUID 형식의 payment_id가 있는지 확인
"payment_id":\s*"[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}"

# total이 0보다 큰지 확인 (1~9로 시작하는 숫자)
"total":\s*[1-9][0-9]*
```

**Python 구현:**
```python
import re
import requests

def check_regex(url: str, pattern: str) -> bool:
    response = requests.get(url, timeout=10)
    if response.status_code != 200:
        return False
    
    return bool(re.search(pattern, response.text))

# 사용 예시
is_healthy = check_regex(
    "https://api.example.com/payments/latest",
    pattern=r'"payment_id":\s*"[0-9a-f\-]{36}"'
)
```

### 방법 3: JSON Path Assertion (가장 강력)

JSON 응답에서 특정 필드의 값을 직접 검사합니다. 가장 정밀하고 신뢰할 수 있는 방법입니다.

**JSON Path 예시:**
```
$.data.total > 0           → 검색 결과가 있어야 함
$.status == "ok"           → status 필드가 "ok"여야 함
$.data.users[0].id != null → 첫 번째 유저가 존재해야 함
$.payment_id != null       → payment_id가 null이 아니어야 함
```

**Python 구현 (jsonpath-ng 라이브러리):**
```python
import requests
from jsonpath_ng import parse

def check_json_path(url: str, path: str, expected_value) -> bool:
    response = requests.get(url, timeout=10)
    if response.status_code != 200:
        return False
    
    try:
        data = response.json()
        jsonpath_expr = parse(path)
        matches = jsonpath_expr.find(data)
        
        if not matches:
            return False
        
        actual_value = matches[0].value
        return actual_value == expected_value
    except Exception:
        return False

# 사용 예시
assertions = [
    ("$.status", "ok"),
    ("$.data.total", lambda v: v > 0),  # 커스텀 조건
]

for path, expected in assertions:
    result = check_json_path("https://api.example.com/health", path, expected)
    print(f"  {path}: {'✅ PASS' if result else '❌ FAIL'}")
```

**FastAPI 헬스체크 엔드포인트에 통합하기:**
```python
from fastapi import FastAPI
from sqlalchemy import text

app = FastAPI()

    }

    try:
        db.execute(text("SELECT 1"))
        status["database"] = "connected"
    except Exception as e:
        status["database"] = "error"
```

이제 CheckAPI에서 JSON Path assertion으로 이렇게 설정할 수 있습니다.

```
$.status == "ok"
$.database == "connected"
$.redis == "connected"
```

HTTP 200을 받더라도 DB나 Redis 연결이 끊어졌으면 즉시 알림이 옵니다.

## 얼마나 자주 발생하나요?

실제 데이터를 보면 놀랍습니다.

- **87%** — HTTP 상태 코드만 체크하는 팀의 비율 (업계 추산)
- **40%** — Silent Failure가 전체 API 다운타임의 원인인 비율
- **평균 47분** — Silent Failure를 사용자 신고 없이 자체 탐지하는 데 걸리는 시간
- **$5,600/분** — 평균 프로덕션 다운타임 비용 (Gartner)

| 방법 | 정밀도 | 설정 난이도 | 적합한 상황 |
|------|--------|------------|------------|
